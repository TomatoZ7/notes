# Deployment: 应用管家

Deployment 部署和管理长期运行、无状态的应用服务，确保其稳定运行并易于更新扩缩。

## 为什么需要 Deployment

我们知道 Pod 的 YAML 有一个 `restartPolicy` 字段，默认值是 `Always`，可以监控 Pod 里容器的状态，一旦发生异常，就会自动重启容器。

但是 `restartPolicy` 只能保证容器正常工作，不能保证 Pod 正常运行，例如 Pod 被误删、Pod 所在的节点断电等意外情况。

同时对于高可用、多实例的场景，如果只使用 Pod，就无法自动化扩容、监控，也就没有利用好 k8s 自动化运维的优势。

所以，和 Job/CronJob 一样，k8s 利用对象组合的思想，创建了一个新的对象 Deployment，来管理 Pod。

## Deployment YAML 文件编写

生成 YAML 模板：

```sh
$ kubectl create deploy ngx-dep --image=nginx:alpine --dry-run=client -o yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  creationTimestamp: null
  labels:
    app: ngx-dep
  name: ngx-dep

spec:
  replicas: 1
  selector:
    matchLabels:
      app: ngx-dep
  strategy: {}
  template:
    metadata:
      creationTimestamp: null
      labels:
        app: ngx-dep
    spec:
      containers:
      - image: nginx:alpine
        name: nginx
        resources: {}
status: {}
```

整体结构跟 Job/CronJob 差不多，多了 `replicas`、`selector` 这两个新字段，它们正式 Deployment 实现多实例、高可用等功能的关键所在。

`replicas` 表示副本数量，可以指定在 k8s 集群里运行多少个 Pod 实例。k8s 还会持续监控 Pod 的运行状态，如果有 Pod 发生意外消失了，数量不满足期望数量，它就会创建出新的 Pod，直至符合期望数量。

`selector` 的作用是帮助筛选出被 Deployment 管理的 Pod 对象，`selector.matchLabels` 定义了 Pod 对象应该携带的 label，它必须和 `template` 里 Pod 定义的 `labels` 完全相同，否则 Deployment 就无法找到要控制的 Pod 对象，apiserver 提示 YAML 格式校验错误无法创建。

和 Job 不同，Deployment 和 Pod 通过在 API 对象的 `metadata` 里加各种标签，解除了 Deployment 和 Pod 的强绑定，把组合关系变成了“弱引用”。Deployment 只负责确保 Pod 对象有足够的副本数量运行。这样做的好处是 Pod 还可以被其他对象管理，例如负责负载均衡的 Service 对象。

## kubectl 操作 Deployment

```bash
> kubectl apply -f deploy.yaml
deployment.apps/ngx-dep created

> kubectl get deploy
NAME      READY   UP-TO-DATE   AVAILABLE   AGE
ngx-dep   2/2     2            2           50s
```

- `READY` 表示运行的 Pod 数量。
- `UP-TO-DATE` 表示当前更新到最新状态的 Pod 数量。当你执行 `kubectl apply -f` 更新 Deployment 时，新 Pod 会逐步替换旧版本，这个值会慢慢增加，直到等于期望值。
- `AVAILABLE` 表示通过可用性检查、当前可正常对外提供服务的 Pod 数量。

```bash
> kubectl get pod
NAME                        READY   STATUS      RESTARTS          AGE
ngx-dep-6b5944c95f-57bz4    1/1     Running     0                 14s
ngx-dep-6b5944c95f-pxqnw    1/1     Running     0                 14s
```

被 Deployment 管理的 Pod 自动带上了名字，命名的规则是 Deployment 名 + Pod Template 哈希 + Pod 随机串。

接下来我们尝试删除 Pod，模拟 Pod 故障场景：

```bash
> kubectl delete pod ngx-dep-6b5944c95f-57bz4
pod "ngx-dep-6b5944c95f-57bz4" deleted
> kubectl get pod
NAME                        READY   STATUS      RESTARTS        AGE
ngx-dep-6b5944c95f-mj5rq    1/1     Running     0               3s
ngx-dep-6b5944c95f-pxqnw    1/1     Running     0               10m
```

可以看到，`57bz4` 后缀的 Pod 确实消失了，但很快又创建出了一个新的 Pod，保证应用实例的数量始终是 YAML 里定义的数量。

除此之外，你还可以使用 `kubectl scale` 随时调整 Pod 的数量，实现动态扩缩容：

```bash
> kubectl scale --replicas=5 deploy ngx-dep
deployment.apps/ngx-dep scaled

kubectl get deploy
NAME      READY   UP-TO-DATE   AVAILABLE   AGE
ngx-dep   5/5     5            5           18m
```

但要**注意**，`kubectl scale` 只是命令式操作，扩缩容只是临时措施。如果应用需要长时间保持一个确定的 Pod 数量，最好还是编辑 YAML 文件，修改 `replicas` 字段，再用 `kubectl apply` 修改对象的状态。

在使用 `kubectl get` 的时候，加上参数 `-l`，使用 `==`、`!=`、`in`、`notin` 就可以筛选出要查找的对象，例如：

```bash
> kubectl get pod -l app=nginx
No resources found in default namespace.

> kubectl get pod -l 'app in (ngx, nginx, ngx-dep)'
NAME                       READY   STATUS    RESTARTS   AGE
ngx-dep-6b5944c95f-mj5rq   1/1     Running   0          2m
ngx-dep-6b5944c95f-pxqnw   1/1     Running   0          12m
```

> 实际上 Deployment 并不会直接创建或管理 Pod，它是通过 管理 ReplicaSet（RS） 来间接管理 Pod 的。
>
> Deployment → ReplicaSet → Pod
